Prompt #7: In a registration system where operations occur sequentially, what implementation strategies ensure deterministic promotion from the waitlist?
Response #7: To ensure deterministic promotion in a sequential system, you must eliminate any "hidden" state or arbitrary choices by the software. Determinism means that given the same input sequence, the system must arrive at the exact same memory state every time.

Here are the primary strategies to achieve this using a list and a queue:

1. The "Single-Source" Trigger

A common mistake is to have multiple functions attempt to "check" if they should promote someone. To remain deterministic, the Promotion Logic should be a dedicated, internal sub-routine triggered only by a successful removal from the Registered List.

The Logic: If (User removed from Registered) AND (Waitlist is not empty) THEN (Pop Head of Waitlist -> Append to Registered).

Why it works: By coupling the promotion strictly to the vacancy event, you ensure the state never "drifts" where a spot is open but the waitlist remains full.

2. Atomic Pointer Management (The "Shift")

If you use a List to represent your Queue, you must decide how to handle the "hole" left behind when the first person is promoted.

Strategy A (Shifting): When the user at index 0 is promoted, every other user in the list moves down one index (1→0, 2→1).

Strategy B (Pointer): The list stays still, but a "Head Pointer" moves from index 0 to 1.

Recommendation: Use Shifting. While it is slightly slower (O(n)), it is more deterministic because the "next" person to be promoted is always at index 0. This prevents bugs where a pointer might get out of sync with the actual list size.

3. Identity-Based Search for Cancellations

When a waitlisted user cancels (Requirement 8), they might be in the middle of the queue.

The Strategy: Use an Identity-based filter. Iterate through the queue and remove the first entry that matches the Unique Identifier.

Deterministic Result: Because you are processing sequentially (C7), you search from index 0 forward. If two users had the same ID (which your C6/C9 prevents), you would always hit the same one first. This preserves the relative order of everyone else.
